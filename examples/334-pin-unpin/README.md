# 334: Pin and Unpin
**Difficulty:** ⭐⭐⭐  
**Category:** Functional Programming  



## Problem Statement

Async state machines generated by `async fn` are self-referential: they contain pointers to their own fields (to hold references across `.await` points). Moving a self-referential struct would invalidate its internal pointers — a memory safety violation. `Pin<P>` prevents such movement. `Unpin` marks types that are safe to move even when pinned. Understanding `Pin`/`Unpin` is essential for implementing custom `Future` types and understanding why `.await` requires `Pin<&mut Future>`.

## Learning Outcomes

- Understand that `Pin<P>` prevents the pointed-to value from being moved
- Recognize `Unpin` as a marker trait for types safe to move even when pinned
- Use `PhantomPinned` to opt out of the automatic `Unpin` implementation
- Understand why `Future::poll` takes `Pin<&mut Self>` — futures may be self-referential

## Rust Application

The `SelfRef` struct demonstrates why pinning is necessary:

```rust
pub struct SelfRef {
    data: String,
    ptr: *const u8,   // Points into `data` — moving self would invalidate this
    _pin: PhantomPinned, // Removes auto-Unpin
}

impl SelfRef {
    pub fn new(s: &str) -> Pin<Box<Self>> {
        let mut boxed = Box::new(Self {
            data: s.to_string(),
            ptr: std::ptr::null(),
            _pin: PhantomPinned,
        });
        // Set ptr after boxing — now safe because Box won't move the allocation
        let data_ptr = boxed.data.as_ptr();
        unsafe { boxed.as_mut().get_unchecked_mut().ptr = data_ptr; }
        Pin::new(boxed)
    }
}
```

## OCaml Approach

OCaml's garbage collector handles self-referential data transparently — GC knows about all pointers and updates them if objects move. OCaml does not need an equivalent of `Pin`:

```ocaml
(* OCaml: self-referential structures work naturally *)
type node = { mutable next: node option; value: int }
let n = { next = None; value = 42 }
n.next <- Some n  (* self-referential: fine in OCaml *)
```

## Key Differences

1. **GC eliminates the need**: OCaml's GC tracks all pointers and updates them on object movement — the problem `Pin` solves doesn't arise.
2. **Zero-cost**: Rust's `Pin` is a zero-cost abstraction — it adds no runtime overhead, only compile-time restrictions.
3. **`Unpin` as escape hatch**: Most types are `Unpin` by default (`i32`, `String`, `Vec`) — `Pin` only restricts `!Unpin` types.
4. **In practice**: You rarely implement `Pin` manually; `Box::pin(future)` and `pin_mut!(local)` (from `pin-utils`) handle most cases.

## Exercises

1. Show that a normal struct (without `PhantomPinned`) is automatically `Unpin` by checking `assert_impl_all!(MyStruct: Unpin)`.
2. Implement a `SelfRef` correctly and verify that the internal pointer remains valid after the initial setup.
3. Write a minimal custom `Future` for a self-referential state machine and implement `poll()` taking `Pin<&mut Self>`.
